Current Status

The Stadium Booking System is actively being developed as a production-ready platform for Bron24, a startup based in Tashkent, Uzbekistan. The platform is evolving beyond a traditional booking application by introducing social media-inspired features, enabling users to discover venues, share experiences, interact with communities, and receive personalized recommendations alongside secure stadium reservations.

The backend is built using Java, Spring Boot, Spring Security, JWT, Hibernate, and PostgreSQL, with ongoing enhancements including real-time notifications, digital payments, role-based management, booking analytics, multilingual support, AI-powered recommendations, and scalable cloud-ready architecture to support future expansion across the European market.

Problem

Traditional stadium booking systems often rely on manual scheduling or disconnected platforms, leading to double bookings, inefficient reservation management, delayed confirmations, and a poor user experience. Limited availability tracking and the absence of secure authentication further reduce the reliability of the booking process.

The objective was to develop a centralized, enterprise-grade booking platform that enables secure user authentication, real-time slot availability, online booking management, payment integration, and scalable REST APIs to streamline stadium reservations for both users and administrators.

Technical Decisions

1. Microservice-Ready Spring Boot Architecture

Adopted a modular Spring Boot architecture with layered design (Controller, Service, Repository) to improve scalability, maintainability, and future microservice migration.

2. Stateless Security with JWT

Implemented Spring Security and JWT Authentication to enable stateless authentication, role-based access control (RBAC), and secure REST API communication.

3. Optimized Database Design

Designed a normalized PostgreSQL schema using Spring Data JPA and Hibernate to efficiently manage users, stadiums, bookings, payments, schedules, and transaction history.

4. REST API & Third-Party Integration

Developed versioned RESTful APIs for seamless frontend, mobile, and third-party integrations, including secure payment gateway and notification services.

5. Enterprise Deployment Pipeline

Containerized the application with Docker, managed dependencies using Maven, and integrated Git/GitHub workflows to support CI/CD, collaborative development, and scalable cloud deployment.

Why?

The project was initiated to modernize traditional stadium reservations by replacing manual booking workflows with a secure, scalable, and cloud-ready platform. It aims to combine real-time booking, digital payments, role-based management, and social engagement features into a single ecosystem, delivering a seamless experience for users, stadium owners, and administrators.

Objective about this Project

To build an enterprise-grade Stadium Booking System using Java, Spring Boot, Spring Security, JWT, Hibernate, and PostgreSQL that provides secure authentication, real-time booking management, payment integration, and a scalable backend architecture while supporting future expansion with social networking features, AI-driven recommendations, and multi-region deployment.

Challenges & Mistakes Encountered

Double Booking Due to Concurrent Requests

What Happened

During peak booking periods, multiple users attempted to reserve the same stadium slot simultaneously, resulting in duplicate bookings for identical time slots.

Initial Assumption

The issue was initially suspected to be caused by delayed frontend updates or client-side caching.

Investigation

Backend request tracing and database logs showed that concurrent booking requests were reaching the server before the reservation record was committed to the database.

Root Cause

The booking workflow lacked proper transaction management and concurrency control, allowing multiple requests to access the same available slot simultaneously .cookie_genration() which I used to create and Implement.

Solution | Lesson

Investigation

Debugging

Booking Conflict Under Concurrent Requests

What Happened

Users booking the same stadium slot simultaneously occasionally resulted in duplicate reservations during peak traffic.

Investigation

Analyzed API logs, SQL transaction history, and concurrent request traces to identify race conditions in the booking workflow.

Root Cause

The booking process lacked proper transaction isolation and database locking, allowing multiple requests to reserve the same slot before the database state was updated.

Fix

Implemented transactional booking using Spring Boot, Hibernate, and pessimistic locking to ensure atomic slot allocation and prevent duplicate bookings.

Lesson

High-concurrency applications require robust transaction management and database locking to maintain data consistency and prevent race conditions.

Debugging

JWT Authentication Failure

Problem

After successful login, some users encountered 401 Unauthorized and 403 Forbidden errors when accessing protected APIs.

Although the JWT token was generated correctly, authenticated users were intermittently denied access to role-specific resources, affecting the reliability of the booking workflow.

First Solution

After successful login, authenticated users occasionally received 401/403 errors while accessing protected APIs.

The JWT token was generated correctly, but role-specific endpoints were inaccessible due to inconsistent authorization and role mapping.

This affected secure access to booking operations and administrative features until the authentication flow was fully optimized.

New Edge Case

Users with valid tokens but different roles (Admin, User, Stadium Owner) were denied access to role-specific endpoints due to inconsistent authority mapping in Spring Security.

Final Solution

The authentication workflow was redesigned by standardizing JWT claims, implementing a custom UserDetailsService, and correctly mapping database roles to Spring Security Granted Authorities. The security configuration was updated to ensure consistent token validation and role-based authorization across all protected REST endpoints.

Additionally, comprehensive integration testing was performed on authentication and authorization flows, including token expiration, invalid token handling, and role-specific access scenarios. This resulted in reliable Role-Based Access Control (RBAC), secure API communication, and a stable authentication system capable of supporting enterprise-scale backend services.

Lesson

A secure authentication system depends not only on valid JWT tokens but also on accurate role mapping, consistent authorization policies, and comprehensive security testing across all protected endpoints.

Limitations

1. High Concurrent Traffic

The current booking engine may require additional scaling mechanisms (e.g., Redis, Kafka, load balancing) to efficiently handle thousands of simultaneous booking requests.

2. Payment Gateway Dependency

Booking confirmations depend on third-party payment gateway availability, which may temporarily affect transaction processing during service outages.

Stripe, Paypal and many other payament methods which i found little bit complex but possible step-wise.

3. Limited Offline Support and Reliability

The platform requires a stable internet connection, with no offline booking or synchronization capabilities currently available.

4. Regional Expansion & Scalability

The system is currently optimized for a limited deployment region and requires localization, multi-currency, and timezone support for global expansion.

5. Notification Infrastructure as per User Friendliness

Real-time notifications are currently limited and can be enhanced with event-driven messaging services for instant booking updates and reminders.
